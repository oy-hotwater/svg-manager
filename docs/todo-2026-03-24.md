# svg-manager システム改善提案書 drafted by Gemini 3.1 pro 2026-03-24

本ドキュメントは、`svg-manager` リポジトリのソースコードおよび構成を厳格に精査し、今後の保守性、拡張性、および堅牢性を向上させるための具体的な改善案をまとめたものです。

- [x] 1. アーキテクチャと責務の分離（リファクタリング）
- [x] 2. TypeScriptの型定義の厳格化
- [x] 3. 入力値のバリデーションとUX向上
- [x] 4. エラーハンドリングの強化
- [x] 5. テスト環境の導入
- [x] 6. CI/CD（継続的インテグレーション・デプロイ）の導入

## 1. アーキテクチャと責務の分離（リファクタリング）

現状の `src/App.tsx` はコード量が肥大化しており、UIの描画ロジックとデータフェッチ、認証などのビジネスロジックが密結合しています。

- **UIコンポーネントの分割**
  - Viewの状態（`list`, `add`, `detail`）に応じて、コンポーネントを `SvgList`, `SvgAdd`, `SvgDetail` 等として独立したファイルに切り出す。
- **カスタムフックの導入**
  - 状態管理と副作用を分離するため、以下のカスタムフックを作成する。
  - `useAuth`: Firebaseのログイン状態の管理、ログイン・ログアウト処理をカプセル化。
  - `useSvgs`: Firestoreの `onSnapshot` によるデータ購読、およびドキュメントの追加・削除処理をカプセル化。

## 2. TypeScriptの型定義の厳格化

型安全性を確保し、予期せぬバグの発生を防ぐため、型定義を厳格に見直す必要があります。

- **`any` 型の排除**
  - `SvgItem` インターフェースにおける `createdAt` プロパティの型定義から `any` を排除する。
- **データ層とUI層の型の分離**
  - Firestoreから取得した直後の生データの型と、フロントエンドで表示用に加工（日時のパース等）した後の型を明確に区別して定義する。

## 3. 入力値のバリデーションとUX向上

ユーザー入力および環境設定に対する検証を強化し、実行時エラーや表示崩れを未然に防ぎます。

- **SVGコードの検証**
  - 新規SVGの追加時、入力値が空文字でないことの確認に加え、文字列が `<svg` で始まり `</svg>` で終わっているか、あるいは `DOMParser` を用いて正常にパース可能かを検証するロジックを実装する。
- **環境変数のアサーション**
  - `src/firebase.ts` において、`import.meta.env` の必須パラメータが設定されていない場合に即座に例外をスローする処理を追加し、環境構築の不備を早期に検知可能にする。

## 4. エラーハンドリングの強化

外部通信（Firebase API）の失敗時におけるユーザーへのフィードバックを改善します。

- **エラー要因に応じたメッセージの提示**
  - 現状の一律のトースト通知と `console.error` への出力を改め、ネットワーク切断、権限エラー（セキュリティルール違反）など、エラーの種別を判別して適切な対応を促すメッセージをUI上に表示する。

## 5. テスト環境の導入

システムの信頼性を担保するため、自動テストの仕組みを構築します。

- **ユニットテストの実装**
  - Vite環境と親和性の高い `Vitest` および `React Testing Library` を導入する。
  - カスタムフック内のデータ加工ロジックや、複雑なコンポーネントの描画に対するユニットテストを記述する。

## 6. CI/CD（継続的インテグレーション・デプロイ）の導入

手動作業によるミスを排除し、コードの品質を強制的に保つためのパイプラインを構築します。開発時にDockerを使用している利点を活かし、CI環境でも同一のコンテナ環境や再現性の高いクリーンな環境で検証を行います。

- **フェーズ1: CI（静的解析とビルド）の導入**
  - GitHub Actions等を設定し、PushおよびPull Request時に以下を自動実行する。
  - `tsc --noEmit` による型チェック。
  - ESLint / Prettier による静的コード解析。
  - `npm run build` によるビルドの正常終了確認。
- **フェーズ2: テストの自動実行**
  - 「5. テスト環境の導入」完了後、CIパイプラインに `Vitest` の実行ステップを追加する。
- **フェーズ3: 自動デプロイの有効化**
  - mainブランチへのマージをトリガーとし、Firebase Hostingへの自動デプロイ（`firebase deploy`）を実行する設定を追加する。CIが通過した正常なコードのみがデプロイされる状態を確約する。
