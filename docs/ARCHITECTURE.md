# システムアーキテクチャおよびインフラストラクチャ仕様

本ドキュメントでは、SVG Managerにおけるインフラストラクチャのコード化(IaC)と、セキュリティおよび開発体験を維持するための設計意図を定義します。

## 1. Firestoreセキュリティルールの動的生成

Firestoreのセキュリティルールにおいて、特定個人のUIDをハードコードすることは、パブリックリポジトリにおけるシークレット管理の観点から不適切です。本プロジェクトでは、デプロイプロセスに動的置換スクリプトを組み込むことでこの課題を解決しています。

### 処理フロー
1. 開発者は `.env` に `FIREBASE_ADMIN_UID` を定義する（Git管理外）。
2. プロジェクトにはプレースホルダーを含む `firestore.rules.template` のみをコミットする。
3. Firebaseデプロイメントの `predeploy` フックにて `scripts/generate-rules.js` が実行される。
4. Node.js 20 の `--env-file` オプションにより環境変数が読み込まれ、完全な `firestore.rules` が一時的に生成・デプロイされる。

これにより、外部ライブラリ(dotenv等)に依存することなく、モダンなNode.jsの組み込み機能のみでセキュアなIaCを実現しています。

## 2. Windows/Docker(Linux)環境における改行コードの統制

ホストOSがWindows（標準改行コード: CRLF）、コンテナOSがLinux（標準改行コード: LF）という構成において、ボリュームマウントによるファイルの同期は、改行コードの不整合によるESLintのエラーやシェルスクリプトの実行障害を引き起こすリスクがあります。

### 解決策
Gitのグローバル設定(`core.autocrlf`)に依存せず、プロジェクトルートの `.gitattributes` にて以下のルールを強制しています。

```text
* text=auto eol=lf
```

これにより、クローンされたOS環境を問わず、リポジトリ内のすべてのテキストファイルがGitによって強制的にLFとして扱われ、コンテナ環境との完全な互換性が保証されます。

## 3. Firebase Authenticationのアクセス制御
個人専用のデータ管理ツールであるため、第三者によるアカウント作成およびデータベースへのアクセスを完全に遮断しています。

- Sign-up無効化: Firebase Console上からIdentity Platformの「新規ユーザー作成」機能を無効化。

- データアクセス制限: データベースのパスを /users/{userId}/svgs/{document=**} に限定し、ログイン中のUIDとパス変数が一致する場合のみ読み書きを許可。
